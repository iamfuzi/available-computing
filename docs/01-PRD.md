# Available Computing —— 产品需求文档（PRD）

> 版本：v0.1 草案
> 日期：2026-05-05
> 状态：需求评审中

---

## 1. 项目概述

### 1.1 项目名称
**Available Computing（可用算力）**——一个聚合多厂商免费 AI 算力的自托管平台。

### 1.2 一句话定位
**「我有一堆厂商的 API Key，你帮我盯着哪些模型还免费可用，并把它们聚合成一个随时能用的算力池。」**

### 1.3 项目背景

当前 AI 模型生态呈现以下特点：
- 全球十几家厂商（Groq、Google、SiliconFlow、DeepSeek、智谱、Cloudflare 等）提供**永久免费**或**长期免费额度**的模型 API
- 各厂商的免费策略**频繁变动**：模型上下线、限流变更、额度调整
- 开发者通常会同时注册多家厂商，但**无法统一管理**，更不知道当前手头到底有多少"零成本"算力
- 现有方案（One API / LiteLLM / New API）聚焦于「网关 + 计费」，**没有以"免费可用"为第一视角**的产品

### 1.4 核心价值主张

| 已有方案 | 解决的问题 | 未解决的问题 |
|---------|-----------|------------|
| One API / New API | 多厂商 API 聚合、转发 | 不区分免费/付费，需手动配置 |
| LiteLLM | SDK 级别的多厂商兼容 | 没有"可用性"看板 |
| OpenRouter | 自动路由到免费模型 | 闭源、需托管 Key、不支持你已有的厂商账号 |
| awesome-free-llm-apis | 人工维护的免费模型清单 | 静态文档，不是实时状态 |

**Available Computing 的差异化定位**：
> **自托管的开源版 OpenRouter Free**，专注于「**自动发现 + 持续监控 + 聚合展示**」你自己注册的各厂商免费算力。

## 2. 目标用户

### 2.1 用户画像

| 用户类型 | 描述 | 核心诉求 |
|---------|------|---------|
| **个人开发者** | 业余做 AI 应用、Side Project、Demo | 零成本调用 AI，不想为玩具项目付费 |
| **独立创业者 / 早期团队** | 在 PoC / MVP 阶段控制成本 | 用免费额度跑通流程，付费再迁移 |
| **AI 学习者 / 学生** | 学习 LLM、Agent、RAG | 大量调用试错，需要稳定的免费来源 |
| **AI 工具自动化用户** | 用 LLM 做信息处理、爬虫摘要、个人助手 | 长期低频调用，免费额度足够覆盖 |

### 2.2 典型场景

**场景 A：开发者每日开机**
> "今天写个新 demo，先打开 Available Computing 看看现在哪些免费模型还活着、速率限制多少，挑一个 Llama 70B 接上。"

**场景 B：模型批量切换**
> "Groq 的 llama-3.3-70b 突然下线了，看 dashboard 发现 SiliconFlow 还有同尺寸的 Qwen2.5-72B，复制 endpoint 一行替换。"

**场景 C：聚合调用**
> "想做一个 agent，不在乎用哪个模型，只要免费可用就行——直接调 Available Computing 的统一接口，让它自动路由。"

**场景 D：发现新算力**
> "Cloudflare Workers AI 上线了新的免费模型，Available Computing 自动探测到并加入了池子，我不用主动关注。"

### 2.3 用户痛点（按优先级）

1. **不知情** —— 不知道当前哪些模型免费可用，每次需手动翻文档/控制台
2. **变动频繁** —— 厂商策略变化快，已配置的模型悄悄收费或下线，调用时才报错
3. **管理分散** —— 多个厂商的 Key、限流、文档、模型名各不相同
4. **缺乏总览** —— 没有一个地方能一眼看清"我现在有多少免费算力"

## 3. 产品目标

### 3.1 北极星指标
**用户通过本产品发出的有效 API 调用次数**（V1.0 前以"每周回访用户数"代替）—— 反映用户真实在用，而非只是看板数字。

> 注：不用"可用免费模型数"作为北极星——该指标容易被多 Channel 重复计数膨胀，且不反映用户是否真正从中获益。

### 3.2 阶段目标

| 阶段 | 目标 | 衡量 |
|------|------|------|
| **V0.1 MVP** | 跑通端到端：配置 Key → 探测 → 展示 | 支持 3 个厂商，能在 dashboard 看到≥10 个免费模型 |
| **V0.5** | 自动化与稳定性 | 定时自动探测、健康检查、可用性历史曲线 |
| **V1.0** | 提供统一调用入口 | OpenAI 兼容 API，自动路由到可用免费模型 |
| **V2.0** | 智能化 | 按任务类型/上下文长度智能选模型，额度预测告警 |

### 3.3 非目标（明确不做）

- ❌ **不做付费模型计费 / 商业网关**——已有 One API、LiteLLM 充分覆盖
- ❌ **不做多租户 SaaS**——定位个人/小团队自托管
- ❌ **不替用户保管 Key**——Key 仅本地存储，不上云
- ❌ **不做模型微调、训练**

### 4.2 F1 — 厂商接入管理

| ID | 需求 | 优先级 |
|----|------|--------|
| F1.1 | 用户可添加厂商，输入 API Key、Base URL（可选）、备注 | P0 |
| F1.2 | 支持每个厂商配置多个 Key（用于轮询/分担限流） | P1 |
| F1.3 | 添加时自动验证 Key 有效性（调一次 `/models`） | P0 |
| F1.4 | 支持启用 / 禁用单个厂商 | P0 |
| F1.5 | Key 在 UI 中默认隐藏，提供查看/复制按钮 | P0 |
| F1.6 | Key 在本地数据库加密存储 | P1 |

**MVP 必须支持的厂商**（首批 3 个）：
1. **Groq** —— 免费模型最多最稳，API 标准
2. **SiliconFlow（硅基流动）** —— 中文友好、模型丰富
3. **Google AI Studio（Gemini）** —— 免费额度大、多模态

**V0.5 扩展**（4-8 个）：DeepSeek、Cloudflare Workers AI、智谱 GLM、月之暗面、阿里百炼

**V1.0 扩展**：HuggingFace Inference、OpenRouter（中转）、Together AI、Cohere、Mistral

### 4.3 F2 — 免费模型探测与发现 ⭐ 核心

#### F2.1 模型列表拉取
- 通过厂商的 `/models` 接口（或等价接口）获取该厂商当前提供的所有模型
- 缓存到本地数据库

#### F2.2 免费判定（多源融合）
按优先级使用以下信号判定一个模型是否"免费可用"：

| 优先级 | 信号 | 说明 |
|------|------|------|
| 1 | 厂商 API 明确返回 `pricing.input == 0` 或 `tag: free` | 如 SiliconFlow 在 model 对象中有 free 标识 |
| 2 | 整厂商免费（Groq、Cloudflare 部分） | 厂商级别标记 |
| 3 | 内置白名单（项目维护） | YAML 文件，记录已知免费模型，定期由社区/维护者更新 |
| 4 | 兜底未知 | 无法判定时标记为 `unknown`，UI 以不同样式展示，由用户决定是否使用 |

**注意：兜底不发探测请求。** 通过"发请求看成功"来判断免费与否是不可靠的——厂商从不在响应体中告知计费情况，只会在账单中体现。

#### F2.2a 免费类型细分
"免费"不是单一概念，需要区分三种情况：

| 类型 | 说明 | 风险 |
|------|------|------|
| **永久免费** | 无 Token 上限，长期可用（如 Groq） | 低 |
| **免费配额** | 有每日/每月 Token 上限，超出后收费（如 Gemini 1500 RPD） | 中——超额会静默收费 |
| **新用户赠送** | 注册赠送一次性额度，用完即止 | 高——用完自动转付费 |

产品必须在 UI 中区分这三种类型，避免用户因误用"免费配额"型模型而产生意外账单。

**白名单结构示例（仅说明，不写实际代码）**：
```yaml
groq:
  all_free: true
siliconflow:
  free_models:
    - Qwen/Qwen2.5-7B-Instruct
    - deepseek-ai/DeepSeek-V2.5
google:
  free_models:
    - gemini-2.0-flash
    - gemini-1.5-flash
    rate_limit: 15/min
```

#### F2.3 模型元数据采集
对每个免费模型，尽可能采集：
- 模型类型（文本 / 多模态 / 代码 / 嵌入）
- 上下文长度
- 速率限制（请求/分钟、Token/天等）
- 提供方原始模型 ID
- 别名（统一展示用）

#### F2.4 定时刷新
- 默认每 6 小时自动重新探测一次（可配置）
- 用户可手动触发"立即刷新"
- 刷新粒度：单厂商 / 全部

### 4.4 F3 — 算力池可视化 ⭐ 核心

#### F3.1 总览看板
顶部统计卡片：
- 已接入厂商数 / 启用中
- 当前可用免费模型总数
- 健康状态分布（绿/黄/红）
- 今日累计调用次数（V1.0+）

#### F3.2 模型列表
- 表格形式展示所有免费模型
- 字段：厂商、模型名、类型、上下文、速率限制、健康状态、最近响应时间、操作
- 支持按 厂商 / 类型 / 健康状态 筛选
- 支持模型名搜索
- 支持排序

#### F3.3 模型列表空状态
当用户尚未添加任何厂商时，列表区域显示空状态引导：
- 居中展示提示文案："还没有接入任何厂商"
- 展示 3 个推荐厂商的快速入口卡片（Groq / SiliconFlow / Gemini），点击直接进入添加流程
- 副文案说明预期效果："添加后系统自动探测免费模型，通常 1 分钟内完成"

#### F3.4 模型详情页
- 模型基本信息，包含免费类型标签（永久免费 / 免费配额 / 赠送额度）
- 免费配额型模型额外显示：每日/每月限额、剩余估算（如可获取）
- 健康状态历史曲线（24h / 7d）
- 该模型的调用日志（V1.0+）
- 一键复制：`base_url` / `model_id` / 完整调用示例（curl / Python / Node.js）
  - **调用示例中自动填入用户的实际 Key**（脱敏显示后几位），而非占位符 `YOUR_KEY`

### 4.5 F4 — 可用性监控

| 需求 | 说明 |
|------|------|
| F4.1 被动健康信号（优先） | 利用用户真实调用的结果更新健康状态，不消耗额外配额 |
| F4.2 主动探测（兜底） | 仅对 **超过 4 小时无真实调用** 的模型发送最小探测请求，频率默认每 2 小时一次（可调） |
| F4.3 状态分级 | 健康（成功 + 响应快） / 慢（成功但响应 > 阈值） / 异常（失败） / 未知（从未探测） |
| F4.4 错误捕获 | 区分：限流 / 鉴权失败 / 模型下线 / 配额耗尽 / 网络错误 |
| F4.5 历史记录 | 保留最近 7 天探测结果用于绘图 |
| F4.6 配额消耗估算 | 显示本产品累计消耗的探测请求数，让用户了解探测开销 |
| F4.7 告警（可选） | 关键模型从健康转异常时，本地通知 / Webhook |

**配额保护原则**：主动探测请求数每天不超过每个模型日限额的 5%。超出保护阈值时自动降频或暂停探测。

### 4.6 F5 — 统一调用入口（V1.0+）

| 需求 | 说明 |
|------|------|
| F5.1 OpenAI 兼容 API | 暴露 `/v1/chat/completions` 等标准端点 |
| F5.2 模型别名 | 用户可调用 `available/free-text-large` 自动路由到当前可用的最大文本模型 |
| F5.3 路由策略 | 随机 / 轮询 / 按响应时间 / 按余额 |
| F5.4 故障转移 | 调用失败自动切换到同类备用模型 |
| F5.5 鉴权 | 本地生成访问 Key 给客户端使用 |
| F5.6 流式输出（必须） | 支持 SSE（`stream: true`），大模型响应必须流式透传，不能等全部生成完再返回 |

### 4.7 F6 — 调用统计与日志

| 需求 | 优先级 |
|------|------|
| F6.1 记录每次调用：模型、用时、token 数、状态 | P1 |
| F6.2 按厂商 / 模型 / 时间维度聚合 | P1 |
| F6.3 估算各厂商免费额度的剩余比例 | P2 |

## 5. 非功能需求

### 5.1 部署形态
- **单机自托管为主**：Docker / 单二进制
- **数据本地化**：SQLite 默认，支持切换到 PostgreSQL
- **零外部依赖**：除了访问厂商 API，不依赖任何云服务

### 5.2 安全
- API Key **本地加密存储**（AES + 用户密码派生密钥）
- Web 界面强制设置访问密码；Docker 部署时支持 `ADMIN_PASSWORD_FILE`（Docker Secret）传入密码，禁止明文密码出现在 `docker-compose.yml` 或命令行历史中
- 不上报任何用户数据到第三方

### 5.3 性能
- Dashboard 首屏加载 < 1s
- 单次模型探测请求超时 ≤ 10s，并发探测以加速整体刷新
- 单实例支持管理 ≥ 50 个厂商、≥ 500 个模型

### 5.4 可扩展性
- 新增厂商通过实现 Adapter 接口，单文件可完成
- 白名单为外置配置文件，可热更新

## 7. 风险与开放问题

### 7.1 风险
| 风险 | 影响 | 应对 |
|------|------|------|
| 厂商免费策略剧烈变动，白名单维护成本高 | 中 | 社区共建白名单，结合自动探测 |
| 厂商 ToS 禁止"聚合代理"使用 | 中 | 仅做单用户自托管，不做转售 |
| 健康探测消耗用户免费额度 | 低 | 探测请求最小化 token，频率可调 |
| AGPLv3 vs MIT —— 借鉴 New API 的 license 限制 | 中 | 参考其架构而非直接 fork，避免协议传染 |

### 7.2 待决策的开放问题
1. **是 fork New API 还是从零开始？** —— 建议「参考架构 + 从零实现」以避开 AGPLv3 传染
2. **MVP 用什么技术栈？** —— Python（开发快、生态好）还是 Go（部署轻、性能好）
3. **白名单如何维护？** —— 自己维护 / 引用 awesome-free-llm-apis / 两者结合
4. **是否做 V1.0 的统一调用接口？** —— 还是只做"算力池展示"，调用让用户自己写
5. **是否需要多用户支持？** —— 个人版只需单用户

## 8. 验收标准（MVP）

满足以下条件即视为 MVP 完成：

1. ✅ 用户可在 Web 界面添加 Groq / SiliconFlow / Gemini 三个厂商的 Key
2. ✅ 添加后系统自动探测并展示该厂商的免费模型
3. ✅ Dashboard 顶部能看到"当前可用免费模型数"统计卡片
4. ✅ 模型列表能正确显示厂商、模型名、类型、状态
5. ✅ 用户能搜索/筛选模型
6. ✅ 用户能一键复制单个模型的调用示例
7. ✅ 部署方式：`docker run` 一行启动，浏览器访问即可
