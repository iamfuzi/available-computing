## 1. 设计原则

### 1.1 设计理念

> **「算力是一种状态，不是一种配置。」**

传统的网关产品（One API / LiteLLM）让用户「配置」哪些模型可用——这是一种静态思维。Available Computing 把模型可用性当作**实时状态**：自动发现、自动监控、自动呈现。用户的角色从「配置者」变成「观察者 + 使用者」。

### 1.2 核心设计原则

| 原则 | 含义 |
|------|------|
| **以"免费可用"为第一视角** | 所有 UI 默认只展示免费且当前可用的模型，付费模型隐藏在筛选项中 |
| **自动发现优先于手动配置** | 用户只配 Key，模型由系统自动找出来；用户**不应该**录入模型名 |
| **状态即展示** | 模型当前是否可用，看一眼就知道，无需点进去查 |
| **透明可信** | 每个"免费"判定结果都能追溯到来源（白名单 / API 字段 / 探测） |
| **可拆卸** | 用户随时可以脱离本产品——直接复制 Key + Endpoint + 模型名即可在原生 SDK 使用 |

### 1.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 部署形态 | 单机自托管，单容器 | 用户的 Key 不出本地，符合"不替你保管 Key"原则 |
| 用户体系 | MVP 单用户，本地密码登录 | 个人工具，复杂用户体系反而是负担 |
| Key 存储 | 本地数据库 + 加密 | 安全底线 |
| 模型来源 | API 拉取 + 白名单融合 | 单一来源不可靠，必须多源交叉 |
| 默认视图 | "可用免费模型"列表，而非"厂商列表" | 强化产品定位 |

## 2. 核心概念定义

设计与文档中频繁使用的术语，统一在此定义。

| 概念 | 定义 |
|------|------|
| **Provider（厂商）** | 一家提供 AI 模型 API 的公司，如 Groq、SiliconFlow、Google |
| **Channel（渠道）** | 用户配置的一条接入：一个厂商 + 一份 Key + 可选 Base URL。一个厂商可以有多个 Channel（多 Key） |
| **Model（模型）** | 一个具体可调用的模型，归属于某个 Channel |
| **Free Model（免费模型）** | 经判定当前可零成本调用的模型；判定依据见 §6 |
| **Health（健康状态）** | 模型当前的可达性与响应质量，三档：🟢 Healthy / 🟡 Slow / 🔴 Down |
| **Pool（算力池）** | 所有 Channel 下、所有"免费 + 健康"模型的并集，是产品的核心呈现对象 |
| **Adapter（适配器）** | 对接单个厂商的代码模块，封装拉取模型列表、判定免费、健康探测的逻辑 |

**实体关系：**
```
Provider (厂商定义，代码内置)
   │ 1
   │ N
Channel (用户配置的接入实例)
   │ 1
   │ N
Model (该 Channel 下发现的模型)
   │
   └── 每个 Model 携带：is_free, health_status, metadata
```

## 5. 关键流程

### 5.1 添加厂商 → 自动发现免费模型

```
用户点击「+添加厂商」
  ↓
选择厂商类型（如 Groq）
  ↓
输入 API Key
  ↓
点击「验证并添加」
  ↓
后端流程：
  1. 调用 Groq /openai/v1/models 验证 Key
  2. 拉取模型列表
  3. 对每个模型，按 §6 判定是否免费
  4. 对免费模型，发起最小探测请求获取健康状态
  5. 写入数据库
  ↓
弹窗显示结果：发现 12 个免费模型 ✓
  ↓
跳转到首页，新模型已出现在列表中
```

### 5.2 定时自动探测流程

```
后台调度器 (每 6 小时触发)
  ↓
对每个启用的 Channel：
  ├─ 重新拉取模型列表（处理新增/下线）
  ├─ 重新判定免费状态（处理收费策略变化）
  └─ 触发一次健康探测
  ↓
更新数据库
  ↓
首页 WebSocket 推送 → UI 自动刷新统计数字
```

### 5.3 健康探测流程

```
对单个免费模型执行健康探测：
  ↓
构造最小请求（如 "hi"，max_tokens=1）
  ↓
带 10s 超时调用
  ↓
分类结果：
  ├─ 成功 + 响应 < 阈值      → 🟢 Healthy
  ├─ 成功 + 响应 ≥ 阈值      → 🟡 Slow
  ├─ 429 限流               → 🔴 Down (rate_limited)
  ├─ 401/403 鉴权失败       → 🔴 Down (auth_failed)
  ├─ 404 模型不存在         → 标记下线
  ├─ 5xx / 网络错误         → 🔴 Down (server_error)
  └─ 超时                  → 🔴 Down (timeout)
  ↓
记录到健康历史表（保留 7 天）
```

### 5.4 用户使用模型的两种方式

**方式 A：直接调用（MVP 默认）**
```
用户在 Dashboard 看到模型可用
  ↓
点击「复制调用示例」
  ↓
粘贴到自己的代码中（用自己的 Key 直连厂商）
  ↓
调用厂商 API
```

**方式 B：通过统一接口（V1.0+）**
```
用户配置代码中的 base_url 为 Available Computing
  ↓
调用 /v1/chat/completions, model="available/llama-3.3-70b"
  ↓
Available Computing 后端：
  ├─ 查询当前可用 Channel
  ├─ 选一个（按策略）
  ├─ 转发请求
  └─ 记录调用日志
  ↓
返回响应给用户
```

## 6. 免费模型判定策略（核心）

这是产品的最核心算法，单独详述。

### 6.1 判定流程（按顺序）

```
对一个 Model 判定 is_free:
  ↓
[Step 1] 厂商级标记
  ├─ 内置数据：Groq、Cloudflare 等整厂商免费 → is_free = true, source = "provider_free"
  └─ 否则继续
  ↓
[Step 2] API 字段
  ├─ 模型对象中有明确字段如 pricing.input == 0、tags 含 "free"
  │   → is_free = true, source = "api_field"
  └─ 否则继续
  ↓
[Step 3] 白名单匹配
  ├─ 模型 ID 命中维护的白名单（如 SiliconFlow 的特定模型）
  │   → is_free = true, source = "whitelist"
  └─ 否则继续
  ↓
[Step 4] 兜底探测（可选，开销大，默认关闭）
  ├─ 发一次最小请求，检查响应头/响应体是否提及计费
  │   → 不可靠，仅作弱信号
  └─ 默认 is_free = false, source = "unknown"
```

### 6.2 透明度要求

UI 上每个被标记为「免费」的模型，悬停 / 点击应能看到判定来源：

```
🟢 免费 ⓘ
   ├─ 来源: 白名单
   ├─ 白名单条目: siliconflow/Qwen2.5-7B (v2026.05.01)
   └─ 上次验证: 2 小时前
```

### 6.3 白名单数据结构（概念，非代码）

存储为 YAML 文件，按厂商组织：

```yaml
version: 2026.05.01
providers:
  groq:
    free_strategy: all              # 整厂商免费
    notes: "Groq 全部模型当前免费，速率限制有差异"

  siliconflow:
    free_strategy: whitelist
    free_models:
      - id: Qwen/Qwen2.5-7B-Instruct
        notes: "永久免费"
      - id: deepseek-ai/DeepSeek-V2.5
        rate_limit: { rpm: 100 }

  google:
    free_strategy: whitelist
    free_models:
      - id: gemini-2.0-flash
        rate_limit: { rpm: 15, rpd: 1500 }
      - id: gemini-1.5-flash
        rate_limit: { rpm: 15 }
```

### 6.4 白名单维护策略

| 问题 | 方案 |
|------|------|
| 谁维护？ | 项目维护者 + 社区 PR |
| 怎么更新？ | GitHub 上的 YAML 文件，应用启动 / 用户点击「检查更新」时拉取 |
| 多久更新一次？ | 计划每月 1 次基线更新，重大变化即时更新 |
| 用户能否覆盖？ | 可以——本地白名单优先级高于远程 |

### 7.2 关键约束
- 同一厂商可存在多个 Channel（多 Key），但 (provider_type, name) 唯一
- Model 不直接存厂商，必须通过 channel_id 关联
- 同一 model_id 在不同 Channel 下视为不同 Model 实例（健康状态分别记录）

## 8. 适配器（Adapter）抽象

为支持快速接入新厂商，定义统一的 Adapter 接口（概念层）：

```
interface ProviderAdapter {
    // 元信息
    getProviderId(): string         // "groq"
    getDisplayName(): string        // "Groq"
    getDefaultBaseUrl(): string

    // 核心能力
    validateKey(key, baseUrl): Result<void, AuthError>
    listModels(key, baseUrl): Result<Model[], Error>
    isFreeModel(model, whitelist): { is_free, source }
    healthCheck(model, key): Result<HealthInfo, Error>
}
```

新增厂商 = 实现一个 Adapter + 在白名单 YAML 中加一节 + 注册到 Adapter Registry。

## 10. 设计中暂未决策的点

文档中保留这些开放点，待评审后定稿：

1. **MVP 是否要做 OpenAI 兼容代理？** 当前定位 V1.0+，但若用户呼声高可能前置

4. **多用户/团队版** 何时考虑？目前明确不做，但需评估是否影响早期数据模型设计
5. **白名单是否引入"过期时间"？** 比如某条目超过 30 天未更新自动转为弱信号
6. **Dashboard 默认排序逻辑** —— 按厂商分组？按响应时间？按上下文长度？需用户研究
